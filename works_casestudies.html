01

原盤発注トリガー判定ツール

製造業ドメインブラウザで試せる在庫最適化
課題 ISSUE
製品出荷の変動に対し、原盤(資材)の発注判断が経験に寄りやすく、欠品・過剰在庫のリスクを事前に見づらかった。
打ち手 ACTION
日別消費量、原盤マスター、安全在庫、リードタイムをもとに、発注期限・必要数・候補を見える化するMVPを試作。
工夫 POINT
①累計到達 ②移動平均×(リードタイム+安全在庫) ③在庫枯渇予測を切替可能にし、原盤マスターを基準に判断条件を整理。
結果 RESULT
属人判断を減らす設計として、発注候補と根拠を画面上で確認できる形に可視化。
HTMLJavaScript営業日計算移動平均/安全在庫マスター駆動設計
工場長の知見が設計に：「在庫を切らさず、かつ持ちすぎない」発注点の考え方は、製造現場の生産管理そのもの。現場判断をロジック化し、確認できる形にする試みです。

02

Power Query × VBA 集計・帳票自動生成

製造業ドメインブラウザで試せる集計・帳票補助
課題 ISSUE
基幹システムから出力されるCSVの集計、クロス集計、日邸別明細、帳票作成が手作業に残りやすかった。
打ち手 ACTION
Power Queryでデータを取り込み・整形し、マスター結合や集計ルールを整理。公開デモでは集計画面をHTMLで再現。
工夫 POINT
抽出ルール・レイアウト・システム設定をテーブルで外出しし、処理手順を再現しやすい形に整理。
結果 RESULT
手作業を再現可能な処理へ置き換える設計として、帳票作成・在庫判断の補助MVPを検証中。
Power Query(M言語)VBAExcelマスター駆動(ルールテーブル)
[ 基幹システム出力 ]
    ↓ Power Query で取込・整形・結合
[ 統合データ ]
    ↓ ルールテーブル(抽出/レイアウト/設定)で制御
    ↓ VBA でリスト・帳票を生成
[ 日次リスト ] [ クロス集計 A/B/C ] [ 印刷用シート ]
スクリーンショット掲載予定※マスタ実データが写り込まないよう加工して掲載
「設定で動く」設計思想：処理をコードに埋め込みすぎず、ルールをテーブルに外出しする。製造現場の作業標準化と同じ発想で、属人判断を減らす設計です。

公開デモは架空サンプルのみで再現しています。実Excel、社内業務マスター、図面、会社固有の品番は公開していません。

03

物件データ整形ツール

副業実務の横展開ブラウザで試せる入力・確認補助
課題 ISSUE
物件データの列構成がバラバラで、確認・転記・入力前チェックに手作業の負担があった。
打ち手 ACTION
列マッピング・除外ルール・印刷レイアウトを設定駆動で定義し、CSVを読み込んで整形できる補助デモを試作。
工夫 POINT
除外条件をAND/OR・優先度付きで定義可能に。A4印刷の用紙幅自動縮小など、現場の実務要件を反映。
結果 RESULT
賃貸不動産実務で学んだ確認・転記業務を、入力支援MVPとして横展開。
HTMLJavaScriptSheetJS(xlsx)設定駆動UI印刷レイアウト制御
別ドメインへの横展開：副業で触れている賃貸不動産実務を題材に、確認・転記・入力前チェックを設定として整理した補助デモです。

04

Google スプレッドシート 集計ツール群(GAS)

設計説明マスター準拠クラウド集計
課題 ISSUE
複数人で扱うデータの集計を、クラウド上で共有しながら確認したいニーズがあった。
打ち手 ACTION
マスター準拠型で、日次別・終別・集計の各処理をGAS(Google Apps Script)で試作。
工夫 POINT
マスターを基準にデータの整合性を保ちつつ、複数の集計ビューを作れる構成を検討。
結果 RESULT
クラウド上での集計確認と共有の補助MVPとして検証中。
Google Apps Scriptスプレッドシートマスター準拠設計クラウド共同編集
スクリーンショット掲載予定※実データを架空サンプルに置き換えて掲載
同じ発想を別環境でも：Power Query/VBA(オンプレ)、JavaScript(ブラウザ)、GAS(クラウド)。環境が変わっても「現場課題を動く仕組みにする」という軸は一貫しています。

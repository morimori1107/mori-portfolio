01

原盤発注トリガー判定ツール

製造業ドメインブラウザで試せる在庫最適化
課題 ISSUE
製品出荷の変動に対し、原盤(資材)の発注タイミングが担当者の経験頼みで、欠品・過剰在庫のリスクがあった。
打ち手 ACTION
日次の使用実績から、3つの発注判定方式を実装。発注期限・必要数を自動算出し、発注候補をCSV出力。
工夫 POINT
①累計到達 ②移動平均×(リードタイム+安全在庫) ③在庫枯渇予測、を品番ごとに切替可能。マスター駆動で設定を一元管理。
成果 RESULT
発注判断の属人化を解消し、欠品リスクを事前に可視化(※実測値を記入)。
HTMLJavaScript営業日計算移動平均/安全在庫マスター駆動設計
工場長の知見が設計に：「在庫を切らさず、かつ持ちすぎない」発注点の考え方は、製造現場の生産管理そのもの。ドメインを知る人間だからこそ、現場で使える判定ロジックを設計できた。

02

物件データ整形ツール

不動産ドメインブラウザで試せるデータ整形
課題 ISSUE
物件データの列構成がバラバラで、エリア別の整形・不要行の除外・印刷用整形に手作業の時間がかかっていた。
打ち手 ACTION
列マッピング・除外ルール・印刷レイアウトを設定駆動で定義し、ワンクリックでエリア別に整形・Excel/CSV出力。
工夫 POINT
除外条件をAND/OR・優先度付きで定義可能に。A4印刷の用紙幅自動縮小など、現場の実務要件を反映。
成果 RESULT
整形作業の標準化と時間短縮(※実測値を記入)。担当者による仕上がりのばらつきを解消。
HTMLJavaScriptSheetJS(xlsx)設定駆動UI印刷レイアウト制御
不動産実務の粒度：「どの列が要るか」「どの行を除外すべきか」は、賃貸管理の実務を知らないと設計できない。現場で本当に必要な整形だけを、設定として外出しした。

03

Power Query × VBA 集計・帳票自動生成

製造業ドメイン設計説明(社内ツール)基幹データ自動化
課題 ISSUE
基幹システムから出力されるデータの集計・帳票作成が手作業で、工数と転記ミスが発生していた。
打ち手 ACTION
Power Queryでデータを取り込み・整形、VBAで日次リスト・クロス集計・印刷用シートを自動生成する仕組みを構築。
工夫 POINT
抽出ルール・レイアウト・システム設定をテーブルで外出しし、コードを触らず設定変更で運用できる構成に。
成果 RESULT
集計・帳票作成工数の削減と転記ミス防止(※実測値を記入)。現職で自主的に推進中。
Power Query(M言語)VBAExcelマスター駆動(ルールテーブル)
[ 基幹システム出力 ]
    ↓ Power Query で取込・整形・結合
[ 統合データ ]
    ↓ ルールテーブル(抽出/レイアウト/設定)で制御
    ↓ VBA で自動生成
[ 日次リスト ] [ クロス集計 A/B/C ] [ 印刷用シート ]
スクリーンショット掲載予定※マスタ実データが写り込まないよう加工して掲載
「設定で動く」設計思想：処理をコードに埋め込まず、ルールをテーブルに外出しした。これは製造現場の「作業標準書(SOP)で誰でも同じ品質を出す」考え方と同じ。属人化を仕組みで防ぐ設計。

本ツールは社内業務マスターを含むため、現物の配布は行っていません。設計と画面で内容をご説明します。

04

Google スプレッドシート 集計ツール群(GAS)

設計説明マスター準拠クラウド集計
課題 ISSUE
複数人で扱うデータの集計を、クラウド上で共有しながら自動化したいニーズがあった。
打ち手 ACTION
マスター準拠型で、日次別・終別・集計の各処理をGAS(Google Apps Script)で自動化するツール群を構築。
工夫 POINT
マスターを基準にデータの整合性を保ちつつ、複数の集計ビューを自動生成。クラウドで共同編集前提の設計。
成果 RESULT
クラウド上での集計自動化と共有の効率化(※実測値を記入)。
Google Apps Scriptスプレッドシートマスター準拠設計クラウド共同編集
スクリーンショット掲載予定※実データを架空サンプルに置き換えて掲載
同じ発想を別環境でも：Power Query/VBA(オンプレ)、JavaScript(ブラウザ)、GAS(クラウド)。環境が変わっても「業務を自動化する道具をつくる」という軸は一貫している。
